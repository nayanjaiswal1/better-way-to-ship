# Infrastructure

## Background Jobs

### FastAPI BackgroundTasks Are Not Production-Ready
`BackgroundTasks` runs in-process and dies with the worker.

### Use a Proper Task Queue
- **ARQ** (async, Redis-backed) — recommended for FastAPI
- **Celery** (more complex, battle-tested)
- **RQ** (simpler, Redis-backed)

### What Belongs in a Task Queue
- Email sending
- Report generation
- Image processing
- Third-party API calls
- Any operation > 100ms

## Database Migrations

### Always Use Alembic
- Track schema changes in version control
- Enable rollback capability
- Apply migrations automatically in CI/CD

### Migration Rules
- Never modify applied migrations
- Review autogenerated migrations before committing
- Test migrations on copy of production data
- Use `alembic upgrade head` in deployment

## Read Replica Routing

### At Scale: Separate Reads from Writes
- Route reads to read replicas
- Route writes to primary
- SQLAlchemy supports replica routing

### Benefits
- Reduces load on primary DB
- Better read scalability
- Improved read latency

## Connection Pooling with PgBouncer

### The Problem
Each Uvicorn worker holds open connections to Postgres. With multiple workers, you quickly exhaust Postgres connection limits (default: 100).

### The Solution: PgBouncer
- Sits between app and Postgres
- Pools connections efficiently
- Reduces connection overhead
- Configurable pool size

## API Versioning

### When to Version
- Breaking changes to response structure
- Removing fields
- Changing field types
- Changing authentication

### Versioning Strategy
- URL path: `/api/v1/`, `/api/v2/`
- Header-based: `Accept: application/vnd.api+json;version=2`
- Deprecation: announce in headers, maintain old version for 6-12 months

### Running Multiple Versions
- Run v1 and v2 simultaneously during transition
- Redirect deprecated endpoints to new ones
- Log usage of deprecated versions

## ID Strategy

### Rules
- **Never expose internal integer PKs** in URLs or API responses — sequential IDs are guessable (`/users/1`, `/users/2`)
- **Never use UUID v4** for PKs — random order causes index fragmentation, slow inserts at scale
- **Never use UUID v7** when ULID does the same job better (shorter, more readable)

### Chosen Approach

| Purpose | Type | Why |
|---------|------|-----|
| Internal DB PK | `BIGSERIAL` integer | Fastest joins, smallest storage (8 bytes), best index performance |
| Public API ID | `ULID` | Time-ordered (no fragmentation), URL-safe, 26 chars vs UUID's 36, readable |
| Short user codes | `NanoID` | Invite links, order references, short URLs — customizable length |

### Base Model

```python
# pip install python-ulid nanoid
from ulid import ULID
from nanoid import generate
from sqlalchemy import BigInteger, String
from sqlalchemy.orm import DeclarativeBase, mapped_column, Mapped

class Base(DeclarativeBase):
    id: Mapped[int] = mapped_column(
        BigInteger,
        primary_key=True,
        autoincrement=True,
    )
    public_id: Mapped[str] = mapped_column(
        String(26),
        default=lambda: str(ULID()),
        unique=True,
        index=True,
        nullable=False,
    )
```

## Optimistic Locking

Prevents lost updates when two users edit the same record simultaneously.

```python
# models/base.py — add version to every mutable model
from sqlalchemy import Column, Integer
from sqlalchemy.orm import DeclarativeBase

class Base(DeclarativeBase):
    version: int = Column(Integer, nullable=False, default=1)

# repositories/user_repository.py
async def update(self, user_id: int, data: dict, version: int) -> User:
    result = await self.session.execute(
        update(User)
        .where(User.id == user_id, User.version == version)  # version must match
        .values(**data, version=User.version + 1)
        .returning(User)
    )
    updated = result.scalar_one_or_none()
    if not updated:
        raise AppConflictError("Record was modified by another user. Please refresh and try again.")
    return updated
```

```python
# core/exceptions.py
class AppConflictError(AppError):
    status_code = 409

    def __init__(self, message: str = "Conflict"):
        super().__init__(message, "CONFLICT")
```

```tsx
// React — send version with every update
function useUpdateUser() {
  return useMutation({
    mutationFn: ({ id, version, ...data }: UpdateUserPayload) =>
      fetch(`/api/v1/users/${id}`, {
        method: 'PATCH',
        credentials: 'include',
        headers: { 'Content-Type': 'application/json' },
        body: JSON.stringify({ ...data, version }),  // include version
      }).then(async r => {
        if (r.status === 409) throw new Error('Someone else updated this record. Please refresh.');
        return r.json();
      }),
  });
}
```

## Soft Deletes

Never hard delete user data — use `deleted_at` timestamp instead.

```python
# models/base.py — add to every model that needs soft delete
from datetime import datetime
from sqlalchemy import Column, DateTime
from sqlalchemy.orm import DeclarativeBase

class SoftDeleteMixin:
    deleted_at: datetime | None = Column(DateTime(timezone=True), nullable=True, default=None)

# models/user.py
class User(Base, SoftDeleteMixin):
    __tablename__ = "users"
    # ...

# repositories/user_repository.py
async def delete(self, user_id: int) -> None:
    """Soft delete — sets deleted_at, never removes the row."""
    await self.session.execute(
        update(User)
        .where(User.id == user_id)
        .values(deleted_at=datetime.now(timezone.utc))
    )
    await self.session.commit()

async def get_by_id(self, user_id: int) -> User | None:
    """Automatically excludes soft-deleted records."""
    result = await self.session.execute(
        select(User).where(User.id == user_id, User.deleted_at.is_(None))
    )
    return result.scalar_one_or_none()

async def list(self, include_deleted: bool = False) -> list[User]:
    query = select(User)
    if not include_deleted:
        query = query.where(User.deleted_at.is_(None))
    result = await self.session.execute(query)
    return list(result.scalars().all())
```

```python
# Alembic migration — add deleted_at to existing table
def upgrade():
    op.add_column('users', sa.Column('deleted_at', sa.DateTime(timezone=True), nullable=True))
    op.create_index('ix_users_deleted_at', 'users', ['deleted_at'])
```

## Type Sharing (Frontend ↔ Backend)

### The Problem
Separate TypeScript and Pydantic types drift apart, causing runtime errors.

### The Solution: Generate Types from OpenAPI
- **openapi-typescript**: generates TS types from OpenAPI schema
- **datamodel-code-generator**: generates Pydantic from JSON Schema
- Run in CI to catch breaking changes

### Pattern
```
FastAPI defines schemas → OpenAPI schema auto-generated →
openapi-typescript generates TS types → Frontend uses exact same types
```

```bash
# Generate TypeScript types from OpenAPI schema
npx openapi-typescript http://localhost:8000/openapi.json --output src/api/types.ts
```
